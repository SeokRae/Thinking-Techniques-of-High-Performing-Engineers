# 1장: 올바른 관찰하기

## 핵심

성능 문제를 해결하는 첫 번째 단계는 **올바르게 관찰**하는 것입니다.

대부분의 엔지니어는 문제를 잘못 관찰합니다:
- 증상을 원인으로 착각한다
- 관찰할 수 없는 것까지 관찰하려 한다  
- 중요하지 않은 지점만 집중 관찰한다

---

## 사례: 팀장 밥의 실수

### 상황
경영진: "밥, 시스템이 너무 느리다고 고객들이 불평해"

### 밥의 반응 (잘못됨)
- "메모리를 늘려야 한다고 했고"
- "CPU가 부족하다고 했어"
- "데이터베이스가 문제라고 했어"

→ 추측만 있고 증거가 없음!

### 올바른 접근

**1단계: 구체적으로 정의**
```
"청구서 생성이 정확히 330초 걸린다"
- 언제: 매월 말
- 누가: 청구 부서
- 얼마: 330초 (목표: 60초)
```

**2단계: 측정**
```
프로파일링 결과:
- 데이터 조회: 100초 (30%)
- 계산: 180초 (55%) ← 가장 크다!
- 문서 생성: 40초 (12%)
- 저장: 10초 (3%)
```

**3단계: 우선순위**
```
가장 큰 것부터:
1. 계산 (180초) 개선
2. 데이터 조회 (100초) 개선
3. 문서 생성 (40초) 개선
```

---

## 관찰의 원칙

### ✓ 올바른 관찰
```
구체성 + 측정 + 목표

"청구서 생성이 330초 걸리고
60초 이내로 줄이고 싶다"
```

### ❌ 나쁜 관찰
```
"시스템이 느리다"
→ 뭐가? 누가? 얼마나? 모두 불명확
```

---

## 체크리스트

성능 문제를 발견했을 때:

```
□ 구체적으로 무엇이 느린가?
□ 정확한 응답 시간을 측정했는가?
□ 언제 발생하는가?
□ 누가 영향받는가?
□ 목표는 무엇인가?
□ 현재 상황을 측정할 방법이 있는가?
□ 개선 후 검증할 방법이 있는가?
```
