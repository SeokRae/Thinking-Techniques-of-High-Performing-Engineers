# 8장: 문제 해결 - 요약

## 📌 핵심 개념

- **4가지 질문 루프**: 얼마나 걸리나? → 왜 그렇나? → 만약 ~한다면? → 그 밖에?
- **데이터-주도 사고**: 숫자와 맥락을 함께 봐야 올바른 판단 (수적 피해 방지)
- **피드백 루프**: 예측 전 개선 → 실제 적용 → 결과 측정 → 반복
- **최고 속도**: (1) 불필요 이벤트 없음(효율) + (2) 이벤트가 하드웨어 속도(대기 없음)
- **대기 시간 숨기기**: 병렬화로 응답 시간은 줄지만 총 부하는 유지
- **과유불급**: 튜닝 파라미터는 욕조 곡선, 최적값을 넘으면 역효과

## 🎓 주요 교훈 (사례 기반)

**엄마의 초단시간 요리**
- 끓는 물을 올려놓고, 기다리는 동안 감자 손질/샐러드 준비
- **대기 시간을 다른 작업으로 숨김**
- 비유: 이것이 병렬화의 핵심

**도미닉의 병렬 실행 데모**
- 오라클 병렬 실행으로 응답 시간 극적 단축
- **하지만 총 작업량(부하)은 그대로**
- 응답 시간 vs 처리량의 트레이드오프

**IT 본부장 더그**
- 요구: 99.999% 가용성
- 현실: 99.99%까지 합리적 비용
- **결정권자와 빠른 피드백 루프** → 즉시 99.99% 결정
- 교훈: 없으면 시간만 낭비

**수적 피해 사례**
- 지표 숫자만 좋아도 사용자 경험이 나쁘면 다시 질문 시작
- 맥락(누가, 언제, 어떤 경로로)을 함께 봐야 함

## 💡 실행 가능한 가이드

### 4가지 질문 반복

1. **얼마나 걸리나?** (스톱워치, 측정)
2. **왜 그렇게 오래 걸리나?** (프로파일, 추적)
3. **만약 ~한다면?** (예측, 시뮬레이션)
4. **그 밖에?** (부수적 영향, 다른 부분에 미치는 영향)

### 임원용 피드백 루프

**적용 전**: "x 방법을 적용하면 얼마나 완화될까?" (예측)
**적용 후**: "x 방법을 적용했더니 얼마나 완화됐나?" (검증)

### 부수적 피해 방지

- 해결책의 범위 ≤ 증상의 범위
- 로컬 설정으로 충분하면 전역 변경 금지
- 우선순위 높은 프로그램이 해당 리소스 부족을 겪는지 먼저 확인
- 통합 테스트로 다른 프로그램 영향 확인

### 최고 속도 판단

**두 조건을 만족하면 최고 속도**:
1. 불필요 이벤트 없음 (효율)
2. 이벤트가 하드웨어 속도로 실행 (대기 없음)

→ 이 이후 개선은 하드웨어 업그레이드만 가능
→ 요구사항이 최고 속도를 초과하면 업그레이드 외 대안 없음

### 과유불급 방지

- 파라미터 조정은 욕조 곡선 (최적 구간 존재)
- 적절한 지점에서 멈추기
- "두 알이면 나아지니 한 병?" 식 과도한 조정 금지

## 📋 체크리스트

- [ ] 4가지 질문을 차례로 물었는가? (측정 근거 확보)
- [ ] 숫자뿐 아니라 맥락을 봤는가? (수적 피해 방지)
- [ ] 임원/리더와 피드백 루프를 닫았는가? (예측 → 실제 검증)
- [ ] 부수적 피해가 없는가? (통합 테스트)
- [ ] 최고 속도 이상의 요구사항은 아닌가?
- [ ] 파라미터 튜닝이 과하지는 않은가? (욕조 곡선)

## 🎯 한 문장 요약

> **성능 문제는 4가지 질문의 반복과 빠른 피드백 루프로 해결하며, 최고 속도를 알아야 올바른 투자 결정을 할 수 있다.**
