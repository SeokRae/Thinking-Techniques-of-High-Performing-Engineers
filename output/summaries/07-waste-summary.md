# 7장: 낭비(Waste) - 요약

## 📌 핵심 개념

- **낭비**: 성능을 갉아먹을 뿐 아니라 비용, 신뢰, 사람 시간까지 잠식
- **제곱 성장 낭비(램프)**: 선형 절감으로 막을 수 없음, **원인 제거** 필수
- **비효율한 프로그램**: 필요 없는 이벤트·호출·출력·데이터 반환 포함
- **효율성의 정의**: 올바른 결과를 **최소한의 이벤트**로 만드는 것
- **보이지 않는 낭비**: 재활용 더미·프린터·화장실 주변에 쌓인 불필요 출력물

## 🎓 주요 교훈 (사례 기반)

**데브라의 청구서 전체 출력 (반복)**
- 원인: 스케줄 작업의 `[마지막 실행 날짜 기억하기]` 체크박스 OFF
- 결과: 130근무일 후 136,240장 중 2,080장만 유효 (98% 낭비)
- 해결: 체크박스 ON → 지난 실행 이후만 출력

**회계 전문가 마사의 불필요 보고서**
- 증상: 화장실 재활용 더미에 "최종 잔액 보고서" 수천 장 폐기
- 진짜 필요: 잔액 불일치 계좌만 찾기 (전체 보고서 아님)
- 조치: 새 애플리케이션에서 불일치 계좌만 보여주고 보고서 비활성화
- 효과: 시스템 CPU 작업량 약 10% 절감 (부하 곡선 왼쪽으로 이동)

**램프 문제의 수식**
- n일째 출력량: `16 × n(n+1)/2` (하루 신규 16장 누적)
- 1년(260근무일): 542,880장 (1,086묶음)
- **제곱 성장은 선형 절감(양면인쇄 등)으로 막을 수 없음**

## 💡 실행 가능한 가이드

### 낭비 찾기와 제거

1. **카운트 제거**: 불필요 작업/출력/쿼리/호출 끄기
2. **설정 확인**: 자동 범위·필터·마지막 실행 시각 등 기본값 점검
3. **성장률 계산**: 제곱 성장 여부를 숫자로 보여주기
4. **현장 관찰**: 사용자 자리·프린터·재활용 더미 등 증상 위치 직접 보기
5. **재발 방지**: 체크리스트·모니터링으로 관리

### 비효율 vs 하드웨어

- **비효율 수정(카운트 감소)** 우선 → 필요시 하드웨어 추가
- 하드웨어 업그레이드는 비효율을 덮지 못하는 경우가 많음
- 마사 사례: 보고서 비활성화 → CPU 10% 절감 (하드웨어 없이)

### 효율성 측정

- 프로파일에서 필요 이벤트 vs 낭비 이벤트 구분
- 낭비 이벤트를 찾아 제거
- "빠르지만 낭비적"인 설계는 비효율

- [ ] 현장 관찰로 보이지 않는 낭비를 발견했는가? (재활용 더미 등)
- [ ] 제곱 성장 여부를 수식으로 계산했는가?
- [ ] 불필요 기능/보고서를 비활성화했는가?
- [ ] 마사 사례처럼 정말 필요한 것만 남겼는가?
- [ ] 낭비 재발을 방지하는 모니터링을 설정했는가?

## 🎯 한 문장 요약

> **낭비는 제곱으로 성장하므로 원인부터 제거해야 하며, 현장에서 눈으로 확인하는 것이 가장 효과적이다.**
