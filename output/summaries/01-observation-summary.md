# 1장: 올바른 관찰하기 - 요약

## 📌 핵심 개념

- **올바른 질문이 먼저**: 답을 구하는 기술보다 **무엇을 물어볼지**가 더 중요
- **질문-답변의 간극**: 기술자가 답할 수 있는 것과 비즈니스가 필요로 하는 것 사이의 차이 인식
- **최적화는 정치적**: 기술만 아니라 조직, 감정, 불안감까지 다루는 문제
- **관찰의 2단계**: (1) 관찰해야 할 것(what)을 파악 (2) 어떻게 관찰할지(how) 정하기
- **증상의 관점**: 사용자가 보고 있는 장면을 직접 관찰해야 진짜 원인을 알 수 있음

## 🎓 주요 교훈 (사례 기반)

**팀장 밥의 4단계 프로세스**
- 경청 → 진정 → 설득 → 해소
- 문제는 기술이 아니라 고객의 불안감과 신뢰 구축
- 성공 사례(과거 경험)가 설득력 있는 무기가 됨

**필리스의 테스트: 진짜 목표 찾기**
- 파일 재배치 = 수단, 작업 속도 향상 = 진짜 목표
- 제안된 해결책이 실제 문제를 푸는지 검증 필수
- 성공/실패가 명확하게 측정되어야 함

**낸시의 칸막이 책상: 증상 현장 방문의 힘**
- 문제: 제록스 청구서 입력 시 "다음" 키를 스물여섯 번 눌러야 함
- 원인 탐색: 서버실은 네트워크/DB 성능을 고민
- 진짜 원인: UI 인터페이스의 비효율 (전체 공급업체 검색 강제)
- 해결책: 접두어 검색 기능 (`x` 입력 후 리턴) → 1초 해결
- 효과: 부서 전체의 생산성 혁신

**교훈: 현장 관찰의 중요성**
- 전화/보고서가 아닌 직접 방문
- 사용자의 책상, 도구, 작업 방식을 봐야 함
- "복도를 돌아다니는 용기" 필요

## 💡 실행 가능한 가이드

1. **증상이 발현된 곳으로 가자**: 서버실이 아닌 사용자 현장으로
2. **사용자의 관점에서 보자**: 기술자의 관점이 아닌 업무 관점에서
3. **물리적 관찰의 가치**: 화면 공유 없이는 놓치는 것들이 있음
4. **직접 방문 불가시**:
   - 화면 공유 (Zoom 등) 활용
   - 텔레메트리 (사용자 행동 측정/기록)
   - 시뮬레이션 (네트워크 지연 등 의도적으로 재현)
5. **리더십의 발휘**: "잘 모르겠습니다. 하지만 문제 해결을 도와드릴 수 있어요."

## 📋 체크리스트

- [ ] 구체적으로 무엇이 느린가? (추상적 표현 피하기)
- [ ] 정확한 응답 시간을 측정했는가?
- [ ] 증상이 발현된 현장을 관찰했는가?
- [ ] 사용자의 진짜 목표(비즈니스 니즈)는 무엇인가?
- [ ] 제안한 해결책이 그 목표를 해결하는가?
- [ ] 개선 후 검증할 방법이 있는가?
- [ ] 사용자의 작업 환경(도구, 데이터, 프로세스)을 파악했는가?
