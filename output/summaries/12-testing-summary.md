# 12장: 테스팅(Testing) - 요약

## 📌 핵심 개념

- **테스트는 프로젝트 끝의 데모가 아님**: 설계·개발 전 과정의 **지속적 경쟁적 과정**
- **위험 개념**: 위험 = 발생 확률 × 비용 (어디서 큰 위험이 숨어 있나?)
- **파괴적 테스트**: 예상 부하를 넘겨 실패할 때까지 밀어붙이기
- **자동화**: 회귀·성능 회귀를 빠르게 감지
- **예방 우선**: "발견"보다 "예방"하는 설계

## 🎓 주요 교훈

**예상보다 큰 위험**
- 데이터, 동시성, 일관성 지연 → 시스템 붕괴
- 재현 가능한 테스트 케이스와 부하/데이터 필요

**위험의 4사분면**
- 가능성 낮음, 치명적: "왼쪽 위" 영역이 중요
- 테스트는 "왼쪽 위(가능성 낮지만 치명적)" 를 드러내는 데 초점

**파괴적 테스트**
- 예상 부하(사용자 수, 데이터 크기)를 넘겨 실패점을 찾기
- 여력(margin) 파악 + 병목·한계 위치 확인
- 좋은 모델 + 파괴적 테스트 = 강력한 예측력

**한 번의 데모는 테스트 아님**
- 프로젝트 끝에 한 번 돌리는 것은 위험
- **개발과 병행해 결함을 찾는 지속적 과정** 필수
- 막판 테스트는 가장 비싼 실수

## 💡 실행 가능한 가이드

### 현실적인 부하로 테스트
- 동시성 (동시 사용자 수)
- 데이터 크기 (대량 데이터)
- 일관성 지연 (네트워크, 캐시)

### 파괴적 테스트 실행
1. 예상 부하에서 시작
2. 부하를 점진적으로 증가
3. 실패하는 지점 기록
4. 여력 = (실패점 - 정상부하) / 정상부하

### 위험 우선순위
- 고비용 + 높은 확률 = 즉시 해결
- 저확률 + 치명적 = 테스트로 검증

### 자동화와 지속성
- 회귀 테스트 자동화
- 성능 기준선 설정
- CI/CD에 성능 테스트 통합

## 📋 체크리스트

- [ ] 현실적인 부하/데이터/동시성으로 테스트했는가?
- [ ] 파괴적 테스트로 실패점을 찾았는가?
- [ ] 위험 우선순위로 테스트 범위를 조정했는가?
- [ ] 자동화 테스트를 설정했는가?
- [ ] 개발 단계에서부터 지속적으로 테스트했는가?
- [ ] 성능 회귀를 감지할 기준선을 확보했는가?

## 🎯 한 문장 요약

> **테스트는 프로젝트 끝이 아닌 전 과정의 지속적 활동이며, 파괴적 테스트로 숨은 위험을 드러내야 한다.**
