# 2장: Method R 방법론 - 요약

## 📌 핵심 개념

- **R 방법론**: Response time(응답 시간) 기반의 성능 개선 체계적 절차
- **문제는 항상 여러 개**: 49개 고충처럼 단일 원인이 아닌 다중 문제 상황
- **우선순위의 중요성**: "무엇이 가장 중요한가?"를 먼저 묻고 그것부터 해결
- **부수적 혜택**: 하나를 고치면 다른 것도 함께 개선되는 경우가 있음
- **은탄환의 함정**: 모든 문제를 한 번에 해결할 방법은 없음

## 🎓 주요 교훈 (사례 기반)

**오렌지 카운티 위기 (1998년)**
- 상황: 매출 수십억 달러의 회사가 주문/배송/반품 모두 정지, 일일 수백만 달러 손실
- 문제: 49가지 고충 목록 (화이트보드 + 스프레드시트 우선순위 정렬됨)
- 팀 배치: 여러 팀이 병렬로 우선순위 높은 항목부터 공략

**배송 송장 출력 문제 (오렌지 카운티 고충 중 하나)**
- 증상: "배송 송장을 출력할 수 없음" (창고 직원들이 박스 앞에서 기다리는 중)
- 잘못된 접근: 대시보드에서 "출력 완료"로 표시됨을 확인 후 끝냄
- 올바른 접근: 출고지 현장으로 직접 방문
- 발견: PC의 FTP 서버가 재시동 후 켜져있지 않음 (하드웨어 문제가 아님!)
- 해결: FTP 서버 1줄 명령 → 송장 출력 재개 → 창고 전체 가동 재개
- **교훈**: 시스템 재시동 후 자동 시작 절차 필요

**핵심 메시지**: 증상이 실제로 발현되는 곳으로 가야 진짜 원인을 찾을 수 있음

## 💡 실행 가능한 가이드

### R 방법론의 7단계

1. **비즈니스 증상을 모두 나열**
   - "보고서가 느려요", "청구서 생성이 느려요", "주문 접수 안 돼요" 등
   - 기술 용어가 아닌 사용자의 관점으로 나열

2. **증상 목록을 비즈니스 우선순위로 재정렬**
   - 영향도(얼마나 많은 사용자?), 손실액(얼마나 큰 피해?), 긴급도(얼마나 급한가?)

3. **우선순위가 높은 증상부터 관찰**
   - 증상이 실제로 발현되는 순간을 직접 봄
   - 화면/대시보드가 아닌 현장 관찰

4. **증상의 원인을 찾기**
   - 관찰 + 측정 데이터로 원인 파악
   - 하드웨어/네트워크/소프트웨어 모두 포함

5. **원인 C에 대한 조치 취하기**
   - 설정 변경, 인덱스 추가, FTP 서버 재시작 등
   - 근본 원인 해결 (임시 패치 아님)

6. **조치 결과 확인**
   - 330초 → 50초 같이 측정 가능한 개선 확인
   - 사용자의 체감 개선 검증

7. **다음 우선순위 증상으로 이동**
   - 49개 중 1개 해결 → 48개 계속

## 📋 체크리스트

- [ ] 증상 목록을 만들었는가? (추상적 "느림" 아닌 구체적 증상)
- [ ] 증상을 비즈니스 우선순위로 정렬했는가?
- [ ] 최우선 증상이 현장에서 실제로 발현되는 것을 확인했는가?
- [ ] 원인을 찾기 위해 데이터를 수집했는가? (추측이 아닌 측정)
- [ ] 원인에 대한 조치 후 증상이 실제로 해결되었는가?
- [ ] 은탄환을 찾으려는 함정에 빠지지 않았는가? (모든 문제의 단일 원인 가정)
- [ ] 우선순위 없이 흥미로운 문제부터 시작하지 않았는가?
- [ ] 부수적 혜택(하나 고치면 여러 개 개선)을 인식했는가?

## 🎯 한 문장 요약

> **성능 문제는 우선순위를 매겨 증상부터 관찰하고 원인을 찾아야 하며, 모든 문제를 한 번에 해결할 은탄환은 없다.**
